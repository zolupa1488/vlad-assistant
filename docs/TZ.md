## 1. Цель и концепция

Построить **личного ИИ-ассистента**, который живёт внутри **обычного Telegram-аккаунта** (не bot-аккаунт через BotFather, а полноценный userbot через MTProto). Снаружи он неотличим от живого человека: имя, аватар, статус, манера речи. Незнакомые собеседники не должны догадываться, что это ИИ.

Ассистент выполняет три роли:

1. **«Второй я» в личке владельца** — отвечает на любые вопросы, опираясь на двухлетний контекст диалогов из ChatGPT, на бизнес-документы (Google Docs) и на статистику коммерческого Instagram. Помогает с маркетингом, цифрами, идеями, формулировками.
2. **Лицо для внешнего мира** — отвечает незнакомцам по ограниченному скрипту в стиле владельца: представляется, отвечает на типовые вопросы, эскалирует нестандартное.
3. **Глаза и уши в групповых чатах** — пассивно сохраняет всю переписку в индекс, чтобы владелец мог в любой момент спросить «что обсуждали в чате X за последние сутки», и автоматически отвечает, когда обращаются по имени или реплаят.

## 2. Ключевые принципы

- **Двухэтапный план поставки.** Этап I (MVP) — обычный Telegram Bot через `@BotFather` + Bot API: быстро запускаемся, отрабатываем мозги, RAG, документы, Instagram. Этап II — миграция на userbot (MTProto) через Telethon, когда у владельца будут на руках `api_id`/`api_hash`. Вся бизнес-логика пишется поверх абстракции `TelegramAdapter`, чтобы переход свёлся к замене одной реализации.
- **Userbot — конечная цель.** Авторизация через личный номер телефона владельца по MTProto. Аккаунт выглядит как обычный человек, а не как «бот с галочкой».
- **Owner-only полные права.** Telegram user ID владельца — единственный, у кого нет ограничений. Все остальные обрабатываются по скриптам и фильтрам.
- **Стиль = стиль владельца.** Системный промпт + few-shot из его переписок. Использует характерные слова, длину сообщений, пунктуацию, эмодзи (или их отсутствие).
- **Ничего «роботизированного» в UX.** Имитация набора (typing indicator), реалистичные задержки перед ответом, разбивка длинного ответа на несколько коротких сообщений, иногда «прочитал и не ответил» (если запрос явно не требует реакции).
- **Память через RAG.** Контекст из ChatGPT, Google Docs и Instagram-инсайтов хранится в векторной БД и подтягивается в нужный момент, а не пихается в промпт целиком.
- **Безопасность данных.** Telegram-сессия и все API-ключи — в секретах (Railway secrets / Fly secrets), никогда в коде/гите.

## 4. Роли пользователей

```
class UserRole(Enum):
    OWNER       = "owner"        # Владимир — полный доступ
    KNOWN       = "known"        # Контакты из whitelist (партнёры, сотрудники)
    STRANGER    = "stranger"     # Все остальные
    BLOCKED     = "blocked"      # Чёрный список — игнор
```

Маппинг ролей хранится в Postgres (`contacts` table). По умолчанию все, кто не в `OWNER`/`KNOWN`/`BLOCKED`, считаются `STRANGER`.

### Что может owner

- Любые вопросы, любая длина, любой контекст.
- Команды действия: «напиши Игорю про оплату», «посмотри в Google Doc такой-то выручку за апрель», «дай дайджест по чату ABC за неделю», «добавь Машу в whitelist».
- Управление поведением: «не отвечай Сергею до завтра», «в чате Y молчи».

### Что может known

- Базовые рабочие переписки в стиле владельца.
- Получение публичной информации о бизнесе (то, что владелец заранее пометил как публичное).
- При нестандартных запросах — пометка «ответ требует владельца», уведомление в личку Володи.

### Что может stranger

- Только сценарный диалог:
  - представление («да, это Владимир, чем могу помочь?»),
  - 5–8 типовых ответов из FAQ (что за бизнес, как связаться по работе, есть ли сейчас наборы/курсы и т.п.),
  - сбор контекста (кто пишет, по какому вопросу) и эскалация владельцу.
- Любая попытка вытащить персональные данные / цифры / документы блокируется и логируется.

## 5. Функциональные требования

### 5.1. Telegram-уровень

- Авторизация через `phone + code + 2FA password` один раз; сессия сохраняется в файл на persistent volume.
- Слушает все апдейты: новые сообщения в личке, в группах, в каналах (read-only), edited, deleted.
- Поддерживает реакции, реплаи, форварды, голосовые (через Whisper), фото (через Vision у Claude).
- Может инициировать сообщения в любом чате, где состоит.
- Соблюдает «человечность»:
  - случайная задержка 2–25 сек перед ответом (зависит от длины),
  - typing indicator на время «думания»,
  - разбивает длинные ответы на 2–4 сообщения с паузами,
  - иногда (по правилам) ставит реакцию вместо ответа.

### 5.2. Слой решений (router)

Перед каждым входящим:

```
1. Определить роль отправителя (owner / known / stranger / blocked).
2. Определить контекст (личка / группа / канал).
3. Применить правила чата (есть ли «молчи здесь», «отвечай только при упоминании»).
4. Решить: отвечать / молчать / только индексировать / эскалировать владельцу.
5. Если отвечаем — собрать контекст (история чата + RAG из памяти) и сходить в Claude.
6. Если эскалируем — отправить владельцу краткую выжимку + кнопки «ответить за меня» / «я отвечу сам».
```

### 5.3. Память (RAG)

Три отдельные коллекции в векторной БД:

- `chatgpt_archive` — диалоги, выгруженные из ChatGPT (см. §8 про формат).
- `business_docs` — содержимое подключённых Google Docs/Sheets, переиндексируется по cron раз в N часов.
- `chat_history` — все сообщения из всех чатов, где сидит ассистент (с метаданными: chat_id, sender, timestamp).

Перед каждым ответом — гибридный поиск (BM25 + dense) по релевантным коллекциям, top-K ≈ 8, передаётся в Claude как `<context>` блок.

### 5.4. Команды владельца (в личке с ассистентом)

Команды распознаются естественным языком — owner просто пишет нормальным текстом, а маршрутизатор определяет интент. Минимальный набор интентов:

- **read_doc** — «глянь выручку за март в доке „Финансы 2026"»
- **chat_digest** — «что было в чате „Партнёры" за вчера»
- **ig_stats** — «сколько охватов было у поста про Х», «топ-5 рилсов за месяц по сохранениям»
- **send_message** — «напиши Игорю, что переносим встречу на 18:00»
- **mute / unmute** — «не отвечай в чате Y до понедельника»
- **add_contact / set_role** — «добавь @nick в known»
- **summarize_unread** — «что я пропустил за ночь»
- **brainstorm** — обычный диалог по маркетингу/идеям, без явного интента.

### 5.5. Скрипт для незнакомцев

YAML-файл `scripts/stranger_script.yaml` с триггерами и ответами. Структура:

```yaml
- intent: greeting
  patterns: ["привет", "здравствуйте", "добрый день"]
  reply_style: warm_short
  template: "Привет! Это Владимир. Чем могу помочь?"

- intent: who_are_you
  patterns: ["кто вы", "чем занимаетесь", "что за бизнес"]
  template: "Занимаюсь {SHORT_BIO}. Если по делу — расскажи, что нужно, разберёмся."

- intent: collab_request
  patterns: ["сотрудничество", "коллаб", "реклама"]
  action: collect_brief
  follow_up: notify_owner
```

Если ни один скрипт не сработал — Claude генерирует ответ в стиле владельца, но строго в рамках общих правил («ничего конкретного по цифрам/документам, эскалируй непонятное»).

### 5.6. Группы

- По умолчанию — пассивный режим: индексация всех сообщений, никаких ответов.
- Реагирует, если: упомянули по имени/нику или реплай на его сообщение.
- Owner может в любой момент написать ассистенту в личку «в группе Y отвечай свободно» / «в Z молчи всегда».
- Раз в сутки (опционально, конфиг) — присылает владельцу дайджест по выбранным «приоритетным» чатам.

### 5.7. Действия с документами

- Подключение Google Drive через service account (владелец расшаривает нужные папки на email сервис-аккаунта).
- При запросе «посмотри в доке X цифру Y» — ассистент:
  1. Ищет документ по названию в индексе.
  2. Читает (используя cached парсинг, обновляющийся по cron).
  3. Извлекает нужную цифру/таблицу/абзац.
  4. Возвращает ответ с цитатой и ссылкой на документ.

### 5.8. Instagram

- Использует **Instagram Graph API** (требует, чтобы аккаунт был Business/Creator и связан с Facebook Page).
- Хранит в БД snapshot статистики постов/рилсов раз в сутки.
- На запрос «сколько было охватов у поста про X» — ищет по подписи/дате/индексированному содержимому, возвращает цифры.
- На «топ-5 рилсов по сохранениям за месяц» — формирует сводку из БД.

## 8. Что нужно от владельца перед первым запуском

1. **Этап I (старт сейчас):** **Bot token** от `@BotFather` — формат `123456789:ABC-DEF...`. Создаётся за минуту в Telegram.
2. **Этап II (потом):** `api_id` + `api_hash` с https://my.telegram.org/apps + **отдельный номер телефона** для userbot-аккаунта (виртуальный/допсимка, не основной — userbot могут забанить за нарушение ToS Telegram).
3. **Аватар, имя, био** для аккаунта.
4. **Свой Telegram user ID** — чтобы зашить как owner. У Владимира: **`401423616`** (положить в Railway secrets как `OWNER_TELEGRAM_USER_ID`).
5. **OpenRouter API key** (https://openrouter.ai/keys) — единый ключ для Claude Opus/Sonnet и любых других моделей. Один ключ, ничего больше не нужно: embeddings считаем локально через `sentence-transformers`.
7. **ChatGPT export (опционально для старта).** Запросить в ChatGPT → Settings → Data Controls → Export Data, дождаться письма с zip-архивом, положить `conversations.json` в `data/chatgpt_export/` и запустить `python -m scripts.ingest_chatgpt`. Без него MVP запускается и работает — RAG над ChatGPT-историей подключится одной командой, когда экспорт придёт.
8. **Google Cloud service account** — JSON-ключ + расшарить нужные доки/папки на email сервис-аккаунта.
9. **Instagram**:
   - Аккаунт переведён в Business/Creator,
   - связан с Facebook Page,
   - получен **Long-lived access token** через Meta for Developers,
   - `instagram_business_account_id`.
10. **Whitelist контактов** — список known с username/phone и ролью.
11. **FAQ для незнакомцев** — 5–10 типовых вопросов и какие ответы давать.
12. **Стилевые примеры** — 30–50 твоих реальных сообщений из переписок (можно копипастом в `scripts/tone_examples.md`), чтобы Claude перенял манеру.

## 9. Этапы разработки (для Claude Code)

### Phase 0 — Каркас в режиме Bot API (день 1)
- Инициализация проекта, Docker, Postgres, Qdrant, Redis локально.
- Создание бота через `@BotFather`, получение токена.
- Подключение **aiogram 3.x**, реализация интерфейса `TelegramAdapter` → `BotApiAdapter`.
- Простейший `on_message` handler: эхо в личке владельцу.
- В `@BotFather` выставить: `/setprivacy → Disable` (чтобы бот видел все сообщения в группах), `/setjoingroups → Enable`.

### Phase 0+ — Подготовка к миграции на userbot (фоновая задача)

## 10. Безопасность и этические оговорки

- **ToS Telegram**: userbot формально нарушает условия. Аккаунт могут заблокировать. Отсюда — отдельный номер и аватар, никакой агрессивной активности (массовые рассылки, инвайты в каналы и т.п.).
- **Privacy**: ассистент индексирует чужие сообщения в групповых чатах. Юридически в РФ это серая зона, морально — стоит понимать риск. Решение владельца.
- **Безопасность ключей**: всё в Railway/Fly secrets. `.session`-файл — в зашифрованном volume. Доступ к серверу — только по SSH-ключу.
- **Anti-leak**: фильтр в `utils/safety.py`, который перед отправкой stranger'у проверяет, не утекают ли в текст имена клиентов, суммы, внутренние процессы. Если да — режет и эскалирует.
- **Логи переписок** хранятся локально в Postgres, не выходят за пределы сервера. Резервные копии — зашифрованы.

## 11. Acceptance criteria (по чему понять, что готово)

- [ ] Незнакомец пишет в личку → получает ответ в стиле владельца, не понимает, что это ИИ; владельцу приходит уведомление с выжимкой.
- [ ] Владелец пишет: «выручка за апрель из дока „Финансы 2026"» → получает корректную цифру с цитатой и ссылкой на документ.
- [ ] Владелец: «топ-5 рилсов за апрель по сохранениям» → таблица из IG.
- [ ] Владелец: «напиши Игорю, перенеси встречу на 18:00» → ассистент показывает черновик, owner подтверждает кнопкой, сообщение уходит Игорю.
- [ ] В групповом чате ассистент молчит, но при «@assistant что думаешь?» отвечает по теме, опираясь на историю чата.
- [ ] Владелец: «что обсуждали в чате „Партнёры" за вчера» → дайджест на 5–10 строк.
- [ ] При перезапуске сервера сессия Telegram восстанавливается без повторной авторизации.
- [ ] Sentry получает ошибки, владельцу приходит алёрт, если бот не отвечает > 5 минут.

## 12. Чего НЕ делать на первой версии

- Не строить веб-админку — всё управление через личку с ассистентом.
- Не делать мульти-аккаунт — один владелец, один userbot.
- Не подключать платежи / финансовые операции от имени владельца.
- Не реагировать на upd'ы из каналов (только чтение по запросу).
- Не делать автоматические массовые рассылки.
